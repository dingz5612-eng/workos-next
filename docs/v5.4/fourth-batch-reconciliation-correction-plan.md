## Needed Reconciliation Migrations

Planned MR-09 migrations should be added through the existing SQL migration
style after `016_checkout_service_process_manager.sql`.

- `reconciliation_import_batches`
  - batch id, tenant/workspace, source type, imported by, source file evidence
    id, source file hash, status, created/closed timestamps.
- `reconciliation_statement_lines`
  - immutable imported line id, batch id, external reference, occurred date,
    amount, currency, payer/payee, channel, memo, normalized hash.
- `reconciliation_matches`
  - statement line to payment/deposit/refund/cash entity mapping, match type,
    amount variance, currency variance, confidence, status, reviewed by.
- `reconciliation_exceptions`
  - unmatched statement, unmatched ledger fact, duplicate match, amount
    mismatch, evidence mismatch, cross-tenant mismatch.
- `reconciliation_decisions`
  - append-only decisions for accept match, reject match, request correction,
    create WorkItem, or waive with reason.
- `reconciliation_runs`
  - runner output, input hash, result hash, counts, grade, and Control Plane
    refs.

Strong constraints:

- statement line hash must be unique per tenant/source.
- a statement line cannot be accepted against multiple money facts.
- accepted match must preserve amount/currency variance fields, not overwrite
  original ledger facts.
- imported files must link to evidence/file records with hash.

## Needed Correction Migrations

- `correction_cases`
  - case id, tenant/workspace, correction type, source object, status, owner,
    reason, risk level, created/closed timestamps.
- `correction_entries`
  - append-only correction event intent, target ledger, target fact id, delta
    amount, currency, references original event/entry, status.
- `correction_approvals`
  - approver, capability, decision, reason, approved/rejected timestamps.
- `correction_impacts`
  - projected effects on payment available amount, deposit held amount,
    stay balance, period metrics, and risk lens.
- `correction_audit_links`
  - links correction case to evidence, reconciliation decision, audit event,
    GateResult/InvariantCheck references.

Strong constraints:

- original Payment, DepositEntry, Charge, DomainEvent, and AuditEvent rows are
  never updated or deleted.
- every correction requires reason, owner, and audit link.
- P0/P1 money corrections require finance/admin approval.
- high-risk corrections must record before/after calculated balances.

## Shadow Strategy

- Capture imported statement batches into `shadow_runtime.compare_inputs` before
  writing formal reconciliation rows.
- Run reconciliation matching in shadow mode against:
  - `hostel_payments`
  - `finance_reconciliations`
  - `payment_allocations`
  - `deposit_transactions`
  - `deposit_liabilities`
  - `hostel_charges`
  - `stay_balances`
- Write proposed reconciliation/correction effects to `shadow_runtime.domain_events`
  and `shadow_runtime.ledger_entries` only.
- Generate `control_plane.shadow_compare_reports` with compare scope:
  - `payment_reconciliation`
  - `deposit_reconciliation`
  - `correction_impact`
  - `stay_balance_rebuild`
- Grade rules:
  - Green: no money mismatch, no duplicate accepted match, all source hashes
    present, all impact previews match recalculated balances.
  - Yellow: display-only column mismatch or low-confidence match left as
    unresolved exception.
  - Red: cross-tenant match, double match, balance mismatch, correction edits
    original fact, missing evidence/hash, shadow consumed by official projector.

  - no direct SQL mutation of money facts.

## Rollback And Compensation Strategy

- Rollback:
  - set `Reconciliation` and `CorrectionCenter` runtime mode to `paused` or
    `legacy`.
  - disable `recon.*`, `correction.*`, and PC reconciliation feature flags.
  - keep imported files, reconciliation batches, decisions, evidence, and audit
    events for traceability.
  - do not delete accepted PaymentLedger, DepositLedger, Charge, or StayBalance
    facts.
- Compensation:
  - incorrect payment allocation correction -> append payment allocation
    reversal/correction event.
  - incorrect deposit correction -> append deposit reversal/correction entry.
  - incorrect refund correction -> append refund failure/cancel/correction
    event as appropriate.
  - incorrect charge correction -> append ChargeCorrectionLite, then request
    StayBalance rebuild.
  - incorrect reconciliation decision -> append new reconciliation decision
    superseding the previous one; preserve the original decision row.
- Control Plane:
  - GateResult must be machine-generated.
  - Red shadow compare blocks active.
  - P0 correction invariant failure blocks active.
  - rollback instruction must be `runtime_mode`.
  - compensation instruction must be `business_reversal` or
    `business_correction`, never SQL delete.

## Blocked Items / Risks

- MR-06 and MR-07 release reports are missing in the current repository.
- Payment and Deposit have runtime storage/policy coverage, but do not yet have
  dedicated command handler classes.
- PC-Finance Lite and PC Deposit pages are not standalone yet.
- No audited statement import/export path exists.
- Evidence attachments store metadata and SHA-256, but no signed URL or
  `file_access_audits` implementation was found.
- Current MR-08 shadow compare evidence is skeleton mode; MR-09 should require
  real reconciliation shadow rows before active rollout.
