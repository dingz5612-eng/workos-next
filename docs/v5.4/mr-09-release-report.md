## Go Evidence

- Bank statements can be imported.
- Import does not generate `PaymentConfirmed`.
- System can generate match candidates.
- Finance can manually confirm a match.
- Mismatches can generate `ReconciliationCase`.
- Correction request requires reason.
- High-risk correction requires approval.
- Correction apply can only append reversal or correction entries.
- Correction after-effects can rebuild StayBalance / DepositBalance.
- PC before/after view is visible.
- `Reconciliation` and `CorrectionCenter` support shadow, pilot, active, and
  rollback routing.

## Rollback

RollbackInstruction:
`docs/v5.4/mr-09-rollback-instruction.json`

Rollback is `runtime_mode` based:

- `Reconciliation` active/pilot -> disabled or legacy
- `CorrectionCenter` active/pilot -> disabled or legacy
- `correction.apply.enabled` off

Forbidden rollback actions:

- delete appended correction
- delete reconciliation decision
- SQL edit old ledger entry
- create `PaymentConfirmed` from bank import

## Compensation

CompensationInstruction:
`docs/v5.4/mr-09-compensation-instruction.json`

Allowed append-only compensation paths:

- correction reversal of correction
- follow-up correction request
- `ReconciliationCase` reopen

Compensation must preserve original ledger entries, original correction audit,
and original reconciliation decisions.

## Residual Risk

The current ShadowCompareReport is generated in skeleton mode and records the
MR-09 grade contract. Production shadow rows must replace the skeleton compare
before active rollout. Full automatic matching remains disabled in active mode.
