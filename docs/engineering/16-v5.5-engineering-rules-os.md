# V5.5 Engineering Rules OS

V5.5 turns the V5.4 Control Plane work into a durable engineering operating
system. The goal is not new business behavior; it is automatic, auditable rule
execution around the Operations Runtime architecture.

## Required Layers

The V5.5 Rules OS is composed of:

- Rule authority: `docs/engineering/00-rule-authority.md`.
- Classified API boundary: `docs/v5.4/operations-api-allowlist.json` until the
  V5.5 classified YAML replaces it.
- Fact ownership registry: planned V5.5 registry for command handler and table
  ownership.
- MR contracts: machine-checkable release contracts for MR-00 through MR-10.
- Invariant maturity model: every invariant declares maturity and target
  maturity.
- GateResult hardening: GateResults are machine-generated, CI-bound, and
  immutable evidence.
- Rule drift detector: OpenAPI, Minimal API, runtime API paths, allowlists,
  ownership, and MR contracts are compared by machine.
- Final Go/No-Go report: generated evidence decides whether the next business
  batch can start.

## Batch Dependencies

- P0 WON-18 Gate Green Fix must be green first.
- A batch defines rule authority and must complete before B, C, D, E, F, or G.
- B and C may run in parallel after A.
- D depends on A, B, and C.
- E may start after C.
- F may run in parallel with E.
- G depends on A, B, C, and D.
- H is final evidence closure only and must not introduce new rules.

## Operations Runtime Direction

All rule layers protect this runtime direction:

```text
Definition -> OperationCase -> WorkItem -> CommandSubmission
-> SliceCommandHandler -> DomainEvent / LedgerEntry
-> ProcessManager -> Projection / Lens -> Mobile / PC Surface
```

Workspace/Card remains a compatibility surface. New APIs, contracts, tests, and
release evidence must target Operations Runtime.
