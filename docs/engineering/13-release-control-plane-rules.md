# Release Control Plane Rules

## Durable Control Plane

The Control Plane seven objects must be physically persisted before a release
can claim V5.4 readiness:

```text
control_plane.release_manifests
control_plane.feature_flags
control_plane.slice_cutover_states
control_plane.shadow_compare_reports
control_plane.runtime_invariant_checks
control_plane.gate_results
control_plane.rollback_instructions
```

GateResult records must be machine-generated by the gate runner or an approved
automation path. Hand-written GateResult evidence is not release evidence.

## Shadow Namespace Isolation

`shadow_runtime` must be physically isolated from formal production facts.
Shadow runner writes go to `shadow_runtime.*`. The official projector must not
read `shadow_runtime.*`, and shadow data must not become a production source of
truth through a convenience query.

## Invariant Monitor Modes

InvariantMonitor output must distinguish:

```text
blocking
observing
```

`blocking` invariants can stop a release or cutover. `observing` invariants can
collect evidence without stopping a release, but their violations must still be
visible in Control Plane reports.

## Rollback Instructions

RollbackInstruction must distinguish:

```text
rollback
compensating
```

`rollback` means moving configuration, runtime mode, or reversible technical
state back. `compensating` means recording a corrective business action when a
business fact cannot be erased safely.

Rollback kinds must stay within the schema allowlist:

```text
feature_flag
runtime_mode
shadow_cleanup
migration_down
business_reversal
business_correction
manual_compensation
```
